써드파티 컨트롤러 데이터 연동 표준 문서

Protocol v3.7
1

개요: 써드파티 컨트롤러 통합의 전략적 가치

인더스트리 4.0 환경에서 제조 데이터의 무결성은 단순히 기록을 넘어 품질 최적화의 핵심 자산입니다. ToolsNet 오픈 프로토콜(Open Protocol) 3.7을 활용하여 써드파티 컨트롤러(Generic Controller)를 통합하는 것은 이기종 장비 간의 '데이터 사일로(Data Silo)'를 제거하고 전사적 차원의 실시간 모니터링 체계를 구축하는 전략적 가치를 지닙니다.

핵심 특징

  • ✓ 표준화된 ASCII 기반 TCP/IP 통신
  • ✓ 하드웨어 제조사 독립적 유연성
  • ✓ 명확한 텍스트 기반 검증 및 유지보수
2

통신 아키텍처 및 논리적 계층 구조

2.1 계층적 식별자 및 고유성

ToolsNet은 System Type, System Number, Unit Number의 조합을 통해 개별 장치를 식별합니다.

Unit (Station)

네트워크에 연결되는 물리적 장치. 하나의 컨트롤러 유닛.

Spindle

유닛 내의 개별 체결 축. 각 유닛 내에서 고유한 번호.

Program

해당 스핀들에서 실행되는 체결 파라미터 세트.

2.2 프로토콜 3.7의 확장성: Multi-unit Socket

버전 3.7 신규 기능: 하나의 시스템 내에서 하나의 소켓을 통해 여러 Unit(Station)이 연결 가능. 네트워크 리소스 절약 및 관리 효율화.

2.3 논리적 구조도

[System (Type + Number)]
    └── [Unit / Station (Number)] ← (3.7부터 단일 소켓 내 다수 Unit 가능)
            └── [Spindle (Number)]
                    └── [Program (Number)]
3

PIM(Protocol Interface Module) 핸드셰이크

3.1 PIM의 전략적 역할

Port 6575를 통한 PIM 핸드셰이크는 시스템 안정성을 위한 필수 관문입니다.

PIM이 제공하는 정보

  • ① 실제 접속할 서버의 IP와 Port
  • ② 마지막으로 성공적으로 저장된 결과 시퀀스 번호

3.2 PIM 핸드셰이크 3단계

단계 전문 명칭 MID 역할
1단계 PIM Verification 12 서버가 PIM 애플리케이션임을 확인
2단계 PIM Info Request 07 컨트롤러 정보 전송
3단계 PIM Information 08 TN IP/Port, 최신 시퀀스 번호 획득
4

데이터 전송: 결과 전문(MID 04) 분석

중요: 써드파티 컨트롤러(System Type 3: Generic)의 모든 전문은 4바이트의 ASCII 길이 정보로 시작해야 합니다.

4.1 MID 04 주요 필드

필드명 길이 포맷 설명
Telegram Length 4 Byte 숫자 이후 데이터의 전체 길이 (0001-9999)
Telegram ID 2 Byte '04' 결과 전문 식별 번호
Identifier 5 Byte 숫자 ACK 매칭을 위한 고유 ID
System Type 4 Byte '0003' Open Protocol Generic 식별
Rundown Seq No 5 Byte 숫자 유닛 내 고유하며 순차 증가
VIN 25 Byte ASCII 차량 식별 번호 또는 바코드

4.2 스핀들 상태 값 정의

Overall Status (1 Byte)

  • 0 OK (정상)
  • 1 NOK (불량)

Torque/Angle Status (1 Byte)

  • 0 OK
  • 1 NOK
  • 2 Low (하한 미달)
  • 3 High (상한 초과)
5

세션 유지 및 통신 안정성 전략

5.1 Keep-Alive 및 Ghost Sockets 방지

MID 06 (Keep-Alive)

전송 데이터가 없어도 30초 주기로 송신

서버 타임아웃

60초간 응답 없는 유닛 연결 강제 종료

5.2 재연결 로직 및 내부 버퍼링

① Transmission Timeout

전문 송신 후 5초 이내에 ACK(MID 05)가 없으면 2회 재전송

② 내부 버퍼링

네트워크 단절 시 모든 체결 데이터를 내부 메모리에 저장. 복구 시 PIM을 통해 획득한 시퀀스 이후부터 순차 재전송

③ Connection Timeout

접속 실패 시 서버 부하 방지를 위해 60초 대기 후 재시도

5.3 시간 동기화

권장사항: Acknowledge 전문(MID 05) 내의 타임스탬프(14 Byte)를 활용하여 컨트롤러의 내부 시계를 정기적으로 동기화할 것을 강력히 권장합니다.

6

구현 체크리스트

성공적인 써드파티 컨트롤러 통합은 System Type 3(Open Protocol generic) 규격의 엄격한 준수와 장애 상황에 대비한 견고한 재시도 로직에 달려 있습니다.

최종 검증 체크리스트

  • 모든 전문 시작 부분에 4바이트 ASCII 길이(Length Header)가 포함되었는가?
  • PIM 핸드셰이크(MID 12, 07, 08)를 통해 동적 포트와 최신 시퀀스 번호를 수신하는가?
  • MID 04 전문 내 Identifier(5 Byte)가 누락되지 않았는가?
  • Rundown Sequence Number가 유닛별로 고유하며 누락 없이 증가하는가?
  • 데이터 부재 시 Keep-Alive(MID 06)가 30초 주기로 발송되는가?
  • 네트워크 장애 시 데이터를 손실 없이 내부 버퍼링하고 재전송하는가?
  • MID 05의 서버 타임스탬프를 통해 컨트롤러 시간 동기화를 수행하는가?
  • (3.7 버전 사용 시) 단일 소켓 내 다수 Unit 운용 시 Unit별 PIM 요청이 개별 수행되는가?
7

ToolsNet 8 UI에서 설정 확인하기

ToolsNet 8 UI 상에서 Open Protocol 3.7 사양서가 올바르게 적용되었는지 확인하려면, 주로 '도구 구조(Tool Structure)'와 '네트워크 맵(Map)' 화면을 확인해야 합니다.

7.1 도구 구조 및 명칭 확인 (Tool Structure)

사양서에 정의된 논리적 구조(Unit, Spindle, Program)가 UI에 어떻게 표시되는지 확인하십시오.

스테이션 이름 (Station Name)

MID 02(Station description) 전문으로 보낸 이름이 UI 상의 해당 유닛 명칭으로 나타납니다.

채널 번호 (Channel Number)

사양서의 '스핀들 번호(Spindle Number)'는 ToolsNet 8 UI에서 '채널 번호'라는 이름으로 표시됩니다.

⚠️ 기본 이름 확인 (Default Names)

만약 UI에 "Station 4", "Spindle 2", "Program 5"와 같이 번호 기반의 이름이 뜬다면, 유닛에서 상세 명칭 전문(MID 01, 02)을 보내지 않고 기본 이름을 사용하고 있는 상태입니다.

7.2 장치 식별 정보 (Administration / Configuration)

관리자 메뉴나 시스템 설정 화면에서 해당 유닛의 등록 정보를 확인하십시오.

고유 조합 확인

ToolsNet 8은 '시스템 타입 번호 + 시스템 번호 + 유닛 번호'의 조합으로 각 장치를 식별합니다.

이 조합이 사양서에 정의된 대로 고유하게 등록되어 있는지 확인이 필요합니다.

시스템 타입 (System Type)

이 화면에서 해당 유닛이 Estic(43번)으로 분류되어 있는지, 아니면 Generic(3번)으로 되어 있는지 확인할 수 있습니다.

7.3 데이터 동기화 및 시퀀스 관리

보고 시작 지점

UI의 결과 리스트에서 체결 데이터가 유실 없이 들어오는지 확인하십시오. PIM 서버는 UI(데이터베이스)에 저장된 마지막 시퀀스 번호를 기준으로 유닛에 데이터 전송 시작 지점을 알려줍니다.

네트워크 맵 (Map View)

ToolsNet 8 UI의 맵 뷰에서는 현재 네트워크에 연결된 모든 유닛의 상태를 시각적으로 확인할 수 있습니다.

UI 확인 포인트 요약

UI 항목 확인 내용 관련 사양서 내용
Tool Structure 유닛/스테이션/스핀들의 계층 구조와 이름 섹션 2 (논리적 계층 구조)
Channel Name 스핀들 번호와 매칭되는 채널 번호 확인 MID 02 (Station Description)
System Type 에스틱(43) 또는 제네릭(3) 등 분류 확인 섹션 2 (계층적 식별자)
Map View 유닛의 연결 상태 및 네트워크 위치 섹션 2 (논리적 계층 구조)

🔍 추가 확인 팁

만약 UI 상에서 특정 유닛의 에러 메시지가 "Unknown Error" 등으로 뜬다면, 시스템 타입이 정확하게 일치하지 않아(예: 43번 대신 3번 사용) 발생한 문제일 수 있습니다.
